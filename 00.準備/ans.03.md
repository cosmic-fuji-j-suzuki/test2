## 重大度：高（着手前に直すべき）

### 1.【最重要】`.gitignore` が存在しない

現在 `.gitignore` が無く、このままコミットすると **本来リポジトリに含めるべきでないものまで追跡対象**になります。

実際に `git status` で以下が未追跡（＝コミット候補）になっています:

```
.claude/        ← 個人のエディタ/権限設定。共有すべきか要判断
00.準備/
CLAUDE.md
CLAUDE_bk.md    ← CLAUDE.md 統合済みのバックアップ。追跡不要の可能性大
```

加えて、CLAUDE.md 自身のポリシーで **`/.trash/` は Git 除外を提案する**と定めており、また将来 `src/` に実装が入れば `bin/`・`obj/`・`vendor/`・`node_modules/` が生成されます。これらを無視する仕組みが今は皆無です。

**対応:** ルートに `.gitignore` を用意する（本回答と同時に **たたき台を作成済み**。後述「今回の対応」参照）。

### 2. Gitのコミットがまだ1件も無い（`No commits yet`）

`git remote` は `origin`（`https://github.com/cosmic-fuji-j-suzuki/test2.git`）が登録済みですが、**ローカルに初期コミットが存在しません**。
CLAUDE.md にはブランチ運用ルール（`feature/…`, `main`, `staging` など）を書きましたが、**`main` にすら中身が無い状態**なので、ルールが機能する前提が整っていません。

**対応（順序）:**
1. `.gitignore` を確定させる（無視対象を決めてからでないと、初回コミットに不要物が入る）
2. `main` に初期コミット（`CLAUDE.md` と `00.準備/` 等のドキュメント）
3. 以降の作業から feature ブランチ運用を開始

> ※ 初期コミット・push は「外部（GitHub）へ反映される取り消しにくい操作」のため、本回答では **実行していません**。ご指示があれば実施します（「今回の対応」参照）。

### 3. CLAUDE.md の命名規則が実ファイルと食い違っている

CLAUDE.md 本文（`## ファイル命名と作業フロー`）は依頼書を **`NN.<slug>.md`（例: `01.init.md`, `03.order.md`）**、回答を **`02.ans.md`, `04.ans.md`** と記載していますが、
実際の `00.準備/` は以下の形式です:

```
user.01.md / ans.01.md
user.02.md / ans.02.md
user.03.md        ← 本依頼
```

つまり実運用は **`user.NN.md`（依頼）→ `ans.NN.md`（回答）** に変わっています。ドキュメントが「正」でなくなると、別セッションのAIが旧命名で生成してしまいます。

**対応（CLAUDE.md を実態に合わせる）:**
- 依頼書: `user.NN.md` / 回答書: `ans.NN.md` と明記
- 「指定がなければ依頼番号を流用して `ans.NN.md`」に統一（`# 出力` 空欄だった本依頼も、この規則で `ans.03.md` としています）
- `現在のやり取りは … 01.init.md→02.ans.md …` の例を現行ファイル名へ更新

## 重大度：中（運用しながらでも可）

### 4. `## git連携` の記述が曖昧・過剰

現状:
```
cosmic-fuji-j-suzuki/test2を利用する。
※ ./.gitを参照。
作業が発生する場合は以下のルールでブランチを作成・コミットする
- feature/{作業番号_作業内容}
- hotfix/{作業番号_作業内容}
- pre-release/{ver}-p-rc
- main
- staging
```

問題点:
- **`※ ./.git を参照` は不適切**。`.git` は内部管理ディレクトリで人が読む場所ではありません。書くならリモートURL（`https://github.com/cosmic-fuji-j-suzuki/test2.git`）を明記。
- **役割の異なるものが同一リストに混在**。`feature/*`・`hotfix/*` は作業ブランチ、`main`・`staging` は常設ブランチ、`pre-release/*` はリリース候補。同じ箇条書きだと関係が読めません。
- **`{作業番号}` の定義が無い**。この連番は `user.NN` の番号か、別採番か。→ 「`user.NN` の番号を流用」など基準を明記すると迷いません。
- **`pre-release/{ver}-p-rc` の `-p-rc`** が意味不明瞭（`rc` = release candidate だとして `-p-` は？）。

**推奨（検証段階に見合った最小構成）:**
検証・学習が主目的なら、GitFlow相当のブランチ群は過剰です。まずは以下で十分:
```
- main           … 常に動く/確定した状態
- feature/NN-slug … 作業単位（NN は user.NN の番号を流用）
（staging / pre-release は、本番デプロイを見据える段階で追加）
```
あわせて **コミットメッセージ規約**（例: Conventional Commits `feat: …` / `docs: …` / `fix: …`）を1行決めておくと、AIのコミット粒度・文言が安定します。

### 5. 「バックアップ除外対象」節が旧構成の名残を含む

`backup/` フォルダを廃止したのに、`### バックアップ除外対象` の本文が
> `backup/` / `mock/` に版を複製する際、生成物・依存物は除外する

と **存在しない `backup/`** を参照しています。Git移行後は「複製時の除外」ではなく **`.gitignore` の除外対象**の話に読み替えるべきです。

**推奨:** 見出しを `### Git 除外対象（.gitignore）` に変え、列挙（`bin/ obj/ vendor/ node_modules/ __pycache__/ .venv/`）はそのまま `.gitignore` の根拠として残す。

### 6. `.claude/` と `CLAUDE_bk.md` の扱いを決める

- `.claude/settings.json` は「このワークスペースの権限設定」。**チームで共有したいなら追跡、個人設定なら除外**。今回は検証用の単独ワークスペースなので **追跡（共有）**を推奨（`acceptEdits` 設定を残す意味がある）。ただし個人トークン等が入る `.claude/` 配下のファイルがあれば要注意。
- `CLAUDE_bk.md` は CLAUDE.md へ統合済みの旧版。**追跡不要**。`.gitignore` に入れるか、`.trash/` へ退避（CLAUDE.md 削除ルール準拠）を推奨。

## Git連携は「問題なさそうか？」への結論

**方向性はOK。ただし “連携できた” 状態には未達**です。現状を正確に言うと:

| 項目 | 状態 | 評価 |
|---|---|---|
| リモート登録（origin） | 済（`test2.git`） | OK |
| ローカル初期コミット | **未（No commits yet）** | 要対応 |
| `.gitignore` | **無し** | 要対応（最優先） |
| ブランチ運用ルール | CLAUDE.md に記載あり | 過剰・曖昧、要整理 |
| 認証（push可否） | 未検証 | 初回 push 時に確認 |

＝「リモートを繋いだ」段階で、「安全に運用を始められる」段階には**あと一歩**です。順序は **①.gitignore 確定 → ②初期コミット → ③push で疎通確認 → ④feature運用開始**。
