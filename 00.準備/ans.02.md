# CLAUDE.md レビュー結果 兼 確認プロンプト対処

対象: `C:\claude\test\test2\CLAUDE.md`（ユーザー修正版）
観点: 実運用に耐えるか / 曖昧さ・矛盾がないか / AIが誤解しない書き方か

---

## 総評

方向性は非常に良いです。「AI駆動開発の検証ワークスペース」という目的、フォルダ構成の事前設計、
`進捗表.md` によるセッション跨ぎの記憶の外部化 —— いずれも `02.ans.md` で説明した勘所を的確に押さえています。

一方で、**運用を始めると必ず詰まる曖昧さ・矛盾**がいくつかあります。以下、重要度順に指摘します。

---

## 重大度：高（着手前に直すべき）

### 1. フォルダ番号と「依頼書の連番」が衝突する

`00.要件定義` `10.設計` … という**フォルダ**の番号体系と、
`01.init.md` `03.order.md` … という**依頼書ファイル**の連番が、同じ「先頭2桁数字」で混在しています。

このままだと、AI も人も「`00` はフォルダ？ ファイル？ どこに置く依頼書？」と迷います。

**推奨（どちらか）:**
- 案A: 依頼書／回答は**ルート直下に連番で置く**、フォルダは**フェーズ用**と役割を明確に分離すると明記する。
- 案B: 各依頼書を**該当フェーズのフォルダ内**に置く（例: `00.要件定義/01.init.md`）。この場合、連番はフォルダ内でリセットするのかルート通し番号なのかを明記する。

現状の `## ファイル命名と作業フロー` は「ルート直下に連番」を前提に書かれているため、**案A を推奨**します。
その旨（「依頼/回答MDはルート直下。番号付きフォルダはフェーズ成果物用」）を1行足すだけで解消します。

### 2. フォルダ番号の付け方が不統一

`00 / 10 / 20 / 30` と10刻みで来たのに、`テスト` と `進捗表.md` には番号がありません。

**推奨:**
- `40.テスト`（テストは開発の後工程なので連番に載せる）
- `進捗表.md` はフェーズに属さない横断ファイルなので、ルート直下のままで良い。ただし
  「番号なし＝全フェーズ横断」というルールを一言添えると、AIが番号採番で迷いません。

### 3. 「Gitを使わない」×「手動バックアップ」の組み合わせは破綻しやすい

`backup/src_ver.N.N.N` `mock/mock_ver.N.N.N` による手動版管理は、**実質的にGitの再発明**です。
検証目的でGitを使わない判断は理解できますが、以下のリスクがあります。

- ソース全体を版ごとに丸ごと複製 → ディスク肥大・差分が追えない
- 「どの版で何を変えたか」がフォルダ名だけでは分からない
- AIが毎回全ファイルコピーを行うとトークン・時間の浪費

**推奨（段階的に）:**
- 最低限、各 `*_ver.N.N.N` に **`CHANGES.md`（変更点の要約）** を必須化する（版名だけでは追えないため）。
- objやvendor等の除外対象を、フォルダ名ベースではなく **除外リスト（`backup-ignore` 的なメモ）** として明記する。
- 検証が進んで本格開発に移る段階では、**ローカルGit（リモートpushなし）への移行**を強く推奨。ローカルGitならAI駆動開発の「差分レビュー」「巻き戻し」が桁違いに楽になります。検証テーマとして「Gitあり/なしの比較」自体が有益。

## 重大度：中（運用しながらでも可）

### 4. `backup` の除外ルールの文面が不明瞭

> `src_ver.N.N.N`(ソースの修正履歴：C#の場合だったら言うobjなどはバックアップしない。PHPの場合も該当するような内容はバックアップしない)

- 「だったら言うobj」は誤記と思われます（「C#なら `bin/`・`obj/`」）。
- 「PHPの場合も該当するような内容」が曖昧。具体的に `vendor/`・`node_modules/` 等を列挙した方が、AIが判断で迷いません。

**推奨（例）:**
```
バックアップ除外: C# → bin/, obj/ ／ PHP → vendor/ ／ JS/TS → node_modules/ ／ Python → __pycache__/, .venv/
```

### 5. 「成果物はMarkdown、途中で決定した開発内容」という文が読みにくい

意図は「①やり取り・意思決定はMarkdownで残す ②実装が進むとPHP/C#/Python等のソースも格納される」だと読めます。
2文に分けると誤読が減ります。

**推奨:**
```
成果物は2種類。
1. Markdown … 要件・設計・意思決定・進捗などのドキュメント
2. ソースコード … 実装が進むと PHP / C# / Python 等が src/ 配下に格納される
```

### 6. `進捗表.md` の追記フォーマットを定義しておくと効く

「追記形式」とだけあると、AIごとに書式がバラつきます。1行テンプレートを決めておくと、セッションを跨いでも一貫します。

**推奨（例）:**
```
## YYYY-MM-DD フェーズ:設計
- 決めたこと: ...
- 未決/宿題: ...
- 次アクション: ...
```

# 【重要】確認プロンプトが出る件について

> Do you want to create 04.ans.md? 1. Yes / 2. ... / 3. No
> 先ほど指定したにもかかわらず出ました。これをやめてください。

## なぜ起きるか（原因）

この確認プロンプトは、**Claude Code の「パーミッション（許可）システム」**が出しているものです。
`CLAUDE.md` や依頼書の「私の確認無しで一気に行ってください」という記述は、
**AI（私）の“進め方の意図”を制御するだけ**で、**ツール実行の許可ゲートは制御できません**。

つまり:

| 層 | 役割 | 「確認無しで」の指示が効くか |
|---|---|---|
| CLAUDE.md / 依頼書の指示 | AIの振る舞い・判断 | 効く（逐次「これでいい？」と聞かない等） |
| パーミッションシステム | ファイル書込・コマンド実行の**許可**| **効かない**（設定で変える必要がある） |

ファイル作成（Write）や編集（Edit）は「取り消しにくい操作」に分類され、
**既定では毎回ユーザーに許可を求める**設計になっています。これは安全装置なので、指示文では抑制できません。

## 止める方法（3つ）

### 方法A: その場で「セッション中は許可」を選ぶ（最も手軽）
プロンプトの **`2. Yes, allow all edits during this session`** を選ぶ、
または **`Shift+Tab`** でパーミッションモードを **acceptEdits（編集自動承認）** に切り替える。
→ そのセッション中はファイル編集の確認が出なくなります。（セッション終了でリセット）

### 方法B: 設定ファイルで恒久的に自動承認（今回こちらを設定しました）
プロジェクトに `.claude/settings.json` を置き、既定モードを `acceptEdits` にすると、
**毎回このワークスペースで編集確認が出なくなります**。今回このファイルを作成しました（下記）。

```json
{
  "permissions": {
    "defaultMode": "acceptEdits"
  }
}
```

- `acceptEdits` は **ファイルの作成・編集のみ**自動承認します。
- 一方で **Bashコマンド（削除・インストール等の危険操作）は引き続き確認**されます。これは意図的に残す方が安全です。
- 「Bashも含め全部確認を消したい」場合は `bypassPermissions` にできますが、**誤操作・破壊のリスクが上がる**ため非推奨です（特に `.trash/` 退避ルールの安全性が損なわれます）。

### 方法C: 特定ツールだけ許可リストに入れる（中間案）
`defaultMode` を変えず、Write/Edit だけ許可する書き方もあります。
```json
{
  "permissions": {
    "allow": ["Write", "Edit"]
  }
}
```

## 今回の対応
上記 **方法B** を採用し、`C:\claude\test\test2\.claude\settings.json` を作成しました。
次回以降、このワークスペースではファイルの作成・編集で確認プロンプトが出ません。
（Bash系の確認は安全のため残します。もし完全に消したい場合はお知らせください。）

> 補足: 設定変更を反映するには、Claude Code の再起動、または `/config` での確認が必要な場合があります。
> もし方法Bで変わらない場合は、方法A（Shift+Tab で acceptEdits）を併用してください。
