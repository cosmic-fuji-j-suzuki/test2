## 0. 用語の整理（先に共通言語をそろえる）  

| 用語 | 意味（このプロジェクトでの定義） |
|---|---|
| グループ | 1 世帯（＝テナント）。データ分離の最小単位。将来の一般展開では「契約単位」に相当 |
| 出納帳 | 月単位の収支一覧（家計簿の中心画面） |
| 費目 | 支出／収入の分類（例：食費、光熱費、給与）。マスタで管理 |
| 貯蓄 | 出納とは別に残高管理する資産。出納と合算して「総資産」を表現 |
| マルチテナント | 1 つのシステムで複数グループのデータを論理的に分離して収容する方式 |

## 1. 目的・背景  

- **対象ユーザー**  
  - 一次対象：個人・家族（親＝家計管理、子＝お小遣い帳）。現状は数人。  
  - 将来対象：一般展開により数千世帯規模の SaaS を想定。  
  → **設計初期からマルチテナント（グループ単位のデータ分離）を前提**とする。  
- **解決したい課題**  
  - 収入に対し支出が多いが、「何が家計を圧迫しているか」が可視化できていない。  
  - 無駄な支出（例：オーバースペックな通信契約）を AI に相談し、プラン見直し等の提案まで受けたい。  
- **提供価値（ゴール）**  
  1. 収支の記録・可視化（どこに・いくら使ったかが一目でわかる）。  
  2. AI による支出診断・節約提案（費目・金額・利用目的から改善案）。  
  3. 貯蓄を含めた総資産の把握。

## 3. 機能要件  

### 3.1 登場人物・権限（RBAC）  

| ロール | 主な権限 | 想定 |
|---|---|---|
| システム管理者 | システム設定、グループ（世帯）の作成・停止、全体運用 | 運営者。将来は課金・契約管理も |
| グループ管理者 | 自グループのマスタ整備（費目・ユーザー等）＋アプリ全機能 | 世帯主など |
| 利用者 | アプリ機能（収支入力・閲覧・AI 相談等） | 家族メンバー |
| （子ども）※要検討 | お小遣い帳（簡易・自分の範囲のみ） | 利用者のサブ種別として実現するか要相談（質問参照） |

- 権限はグループ内で完結（他グループのデータは一切見えない）。  
- 監査観点：作成者・更新者・日時を記録する方針（改ざん・誤操作の追跡用）。  

### 3.2 収支管理（コア機能）  

- 収支データの登録・編集・削除（日付／収支区分／費目／内容／金額／備考）。  
- 月次の出納帳表示（残高計算、月またぎの繰越）。  
- 取込区分の記録（手入力／CSV／レシート）で入力経路を区別。  

### 3.3 レシート取り込み（入力補助）  

- 写真・画像から支出内容を読み取り、入力フォームに補完（OCR）。  
- 位置づけは「入力補助」。読み取り結果はユーザーが確認・修正して確定。  
- ※ OCR エンジンの選定・コスト・精度は要相談（質問参照。無料前提と外部 API コストが競合しうる論点）。  

### 3.4 CSV 取込  

- フロー：**テンプレート DL → 記入 → アップロード → インポート**。  
- 取込時のバリデーション（費目の存在確認、日付・金額形式、重複検知）とエラー明示。  
- 文字コード・区切りは日本語 Excel を想定（UTF-8 BOM／カンマ区切りを基本、要確認）。  

### 3.5 AI 相談  

- 支出項目を分析し「無駄」を指摘、節約策をチャットで相談。  
- 分野横断（通信・保険・光熱等）の見直し提案。  
- アドバイス履歴を保存し、ダッシュボードで参照。  
- ※ 利用する AI（Claude API 等）・コスト・個人の家計データを外部 API に送る際のプライバシー方針は要相談（質問参照）。  

### 3.6 貯蓄・総資産  

- 費目に「貯蓄」を用意し、貯蓄残高を出納とは別に管理。  
- 出納残高＋貯蓄残高＝**総資産**として集計表示。  
- データ設計は「収支データで表現可能か／貯蓄を別テーブルにするか」を P2 で確定（現時点の推奨は質問参照）。  

### 3.7 ダッシュボード（ガジェット）  

- ガジェットボックスを自由配置。追加・削除可。  
- ガジェット案（提案）：  
  - 収支推移（折れ線）／費目別内訳（円・棒）  
  - 今月の予算 vs 実績（※予算機能の要否は質問参照）  
  - 総資産サマリ（出納＋貯蓄）  
  - AI アドバイス履歴／直近の警告（使いすぎ費目）  
  - 固定費一覧（通信・保険など見直し候補）

## 4. 非機能要件  

- **利用者数／同時接続**：現状 2 名程度。将来は数千世帯 → **水平にスケールできる構成**（ステートレス化・DB 分離）を初期方針に含める。  
- **認証・権限**  
  - 3 ロール（システム管理者／グループ管理者／利用者）。  
  - 管理者系は多要素認証（MFA）を検討（TOTP 想定）。  
  - 利用者は「メールアドレス＋パスワード」運用が妥当か後日確定（ログインはグループコード＋ユーザー ID の併用も検討中）。  
- **対応環境**  
  - PC：Microsoft Edge / Google Chrome。  
  - スマホ：主要ブラウザ（レスポンシブ対応）。  
- **性能・可用性**：明確な数値目標なし。「体感で遅い」場合に改善。SLA は将来の一般展開時に定義。  
- **セキュリティ**  
  - パスワードはハッシュ化保存（Laravel 既定の bcrypt/argon2）。  
  - グループ間のデータ分離を厳守（テナント越境アクセス防止）。  
  - 通信は HTTPS 前提。  
  - 家計データ・レシート画像の保管とプライバシー配慮（外部 AI/OCR 連携時は特に）。  
- **国際化（i18n）**：画面文言を JSON 文言ファイルで管理し、言語切替可能に。  
- **テーマ（CSS 外出し）**：配色・デザインを CSS で外部化し、切替可能に（将来のキャラクターコラボ等）。

## 5. 画面  

### 5.1 画面一覧（初期）  

1. ログイン画面  
2. ダッシュボード  
3. 出納帳（月次一覧・入力）  
4. マスタメンテ各種（費目・ユーザー・グループ 等）  
※ 以降、フェーズで段階的に追加。  

### 5.2 共通レイアウト（ログイン後）  

- **ヘッダー**  
  - 1 段目：アプリロゴ・バージョン・ユーザーアイコン（クリックで詳細参照／設定／ログアウトのコンテキストメニュー）。  
  - 2 段目：タブ型メニュー（例：ダッシュボード｜出納帳｜…）。  
- **フッター**：その画面のアクション（例：新規追加｜CSV 出力｜…）。  
- **トースト通知**：画面上部。赤＝エラー／黄＝警告／青＝情報・成功。  
- **チャットボット**：右下固定（アイコン化⇄オープン、最小化可）。実装可能であれば対応。  

### 5.3 主要画面  

- **ログイン画面**  
  - 入力：グループコード（未入力時はシステム管理者扱い）、ユーザー ID、パスワード。  
  - 「情報を記録する」チェック（Remember me。Cookie を基本に検討。要確認）。  
  - ボタン：ログイン、QR 読み取り（カメラ起動しグループ・ユーザーを読取）。  
  - インフォメーション領域（更新履歴等）。  
- **ダッシュボード**：ガジェット配置。フッターアクション：新規ガジェット追加。  
- **出納帳**  
  - ヘッダー：`yyyy年MM月`（`<` `>` とカレンダーで切替、既定は当月）。  
  - 本文：左右分割（右ペインは `>>>` `<<<` でスライド表示・非表示）。  
    - 左：一覧表（日・収支・内容・費目・収入金額・支出金額・残高・備考）。PC は 1 エリアに収めスクロール最小化。  
    - 右：総資産・手元残高・アドバイス（消費特化）。

## 6. データ項目（論理エンティティ・ドラフト）  

現時点の理解での論理モデルです。物理設計（型・制約・インデックス）は P2 で確定します。  

- **グループ**：id, グループ名, 連携コード（将来用）, 状態（有効/停止）, 備考, 監査項目  
- **ユーザー**：id, グループ id, ログイン ID, パスワード（ハッシュ）, 氏名, 氏名カナ, メール, 権限, 状態, 監査項目  
- **費目マスタ**：id, グループ id（共通/グループ別かは要相談）, 収支区分（収入/支出/貯蓄）, 名称, 説明, 表示順, 備考  
- **収支データ**：id, グループ id, 年月日, 収支区分, 費目 id, 内容, 収入金額, 支出金額, 取込区分（手入力/CSV/レシート）, 登録ユーザー id, 備考, 監査項目  
- **貯蓄データ**（要否は要相談）：id, グループ id, 年月日, 貯蓄金額, 関連費目/口座, 備考  
- **AI 相談履歴**（追加提案）：id, グループ id, ユーザー id, 日時, 相談内容, 回答, 関連費目  
- **レシート**（追加提案）：id, 収支データ id, 画像パス, OCR 結果, 取込日時  

> 追加提案の理由：AI アドバイス履歴（3.5）とレシート画像（3.3）は画面要件に登場するため、エンティティ化しておくと漏れが減ります。

## 7. 制約・前提  

- **技術スタック**  
  - 言語：PHP／フレームワーク：Laravel。  
  - DB：お任せ（無料かつセキュリティ担保）。→ **推奨は PostgreSQL または MySQL/MariaDB**（いずれも OSS・無料。Laravel と親和性高。質問参照）。  
  - **Docker** で構成（Web／DB を OS へ直接セットアップしない）。開発環境も Docker で統一。  
- **予算・期限**：予算なし（サードパーティも無料前提）。期限なし（非稼働日・隙間時間で学習を兼ねて進行）。  
  - ※注意：レシート OCR・AI 相談は外部 API 利用時に費用が発生しうる。「完全無料」を優先するか、無料枠/低コストの範囲で機能を実現するかの方針確認が必要（質問参照）。  
- **外部連携**：現状なし。  
- **その他**  
  - レスポンシブ必須（スマホで無理なく操作・閲覧）。  
  - CSS 外出しでテーマ切替可能に。  
  - 文言 JSON 切替で多言語対応（海外展開見据え）。

## 8. 現時点の技術方針（提案・たたき台）  

質問への回答前の暫定案です。合意後に確定します。  

- **アーキテクチャ**：Laravel（モノリス）＋ Blade もしくは Laravel + API + フロント（Vue/React）。モック段階の学習コストを考えると、まずは **Blade + 少量の JS**（または Laravel Breeze/Jetstream）が着手しやすい。  
- **DB**：PostgreSQL を第一候補（型の堅牢さ・JSON 対応）。MySQL に慣れていれば MySQL でも可。  
- **マルチテナント**：単一 DB・`group_id` によるテナント分離（Row Level）を初期採用。規模拡大時に DB 分離へ発展可能。  
- **認証**：Laravel 標準セッション認証＋管理者 MFA（TOTP）。  
- **i18n**：Laravel の言語ファイル（`lang/`）＋必要に応じ JSON。  
- **テーマ**：CSS 変数（カスタムプロパティ）でテーマ切替。

## 9. リスク・論点（先に共有）  

1. **「無料」と外部 AI/OCR コストの競合**：AI 相談・レシート OCR は有料 API が一般的。無料枠 or ローカル OSS（精度低下の可能性）で妥協するかが要判断。  
2. **家計データの外部送信**：AI 相談で個人の家計情報を外部 API に送る是非・匿名化。  
3. **マルチテナントの分離漏れ**：`group_id` フィルタ漏れは情報漏えいに直結。設計・テストで重点管理。  
4. **お小遣い帳（子ども）の扱い**：一般利用者と同一データモデルで足りるか、簡易 UI が要るか。
