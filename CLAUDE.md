# CLAUDE.md  

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.  

## このワークスペースの性質  

Claude Code を利用した **AI駆動開発の検証** を目的としたワークスペースです。  
現時点ではソースコード・ビルド定義・テストは存在せず、成果物は次の2種類:  

1. **Markdown ドキュメント** … 要件・設計・意思決定・進捗などのやり取り  
2. **ソースコード** … 実装フェーズに入ると `30.開発・実装/src/` 配下に PHP / C# / Python 等が格納される  

ビルド・lint・テストのコマンドはまだ定義されていない。  
実装言語・フレームワークが決まった時点で、この節に開発コマンド（ビルド／単体テスト実行を含む）を追記すること。  

## git連携  

- リモート: https://github.com/cosmic-fuji-j-suzuki/test2.git  
- 検証段階につきブランチは最小構成とする:  
  - `main` … 常に動く/確定した状態を保持  
  - `feature/{NN}-{slug}` … 作業ブランチ。`main` から作成し `main` へマージ。  
    `{NN}` は対応する `user.NN` の番号を流用する  
- `staging` / `hotfix` / `pre-release` は、本番デプロイを見据える段階になってから追加する  
- コミットメッセージは Conventional Commits を基本とする（例: `feat:` `fix:` `docs:` `chore:`）  

### push の運用  

- **push はユーザーが手動で実行する。** AI はコミットまでを行う（push はパーミッションで拒否されるため）。  
- push が必要なタイミングで、AI はユーザーに次の実行を案内する:  
  「`! git push -u origin {作業中のブランチ名}` を実行してください」  
- 例: main で作業中なら `! git push -u origin main`、feature ブランチなら `! git push -u origin feature/05-xxxx`。  

## ファイル命名と作業フロー（最重要）  

このワークスペースの運用は、**連番付き Markdown による「依頼 → 回答」の対応関係**で成り立つ。  

- **依頼書**: `user.NN.md`（ユーザーからのオーダー）  
  - 冒頭が `# 役割` / `# 概要` / `# 依頼` / `# その他` / `# 出力` などの見出しで構造化されている  
  - `# 出力` に回答ファイル名を明示できる（省略可）  
- **回答書**: `ans.NN.md`（AI の回答）。`# 出力` に指定があればそれを優先し、無ければ依頼書と同じ番号 `NN` を流用する  
- ユーザーへの質問・回答は末尾に `# ユーザーへの質問・回答` としてまとめる（本文中に散らさない）。記述者によって意味が変わる:  
  - **AI が回答書（`ans.NN.md`）に記述** → ユーザーへの「質問」  
  - **ユーザーが依頼書（`user.NN.md`）に記述** → 直前の AI の質問への「回答」  
  - AI が記述する際は、質問である旨（回答待ちである旨）を明記すること  
- **連番はやり取りの順序**を表す。新しい依頼を処理するときは、まず対象の `user.NN.md` を読み、`# 出力` 指定に従って成果物を作る。  
- **依頼書・回答MDは当該作業フォルダ内に連番で置く。番号付きフォルダはフェーズ成果物用**で、両者の番号体系は別物として扱う（混同しない）。  

現在のやり取りは `00.準備/` 配下（`user.NN.md`→`ans.NN.md`）で進行中。  

## フォルダ構成（予定）  

フェーズを表す番号付きフォルダは10刻み。番号なしファイルは全フェーズ横断。  

- `00.準備/`：  
  最初の相談・方針決め（現在ここ）  
- `01.要件定義/`：  
  基本要件、AIの質問、ユーザーの回答  
- `10.設計/`：  
  環境設計・画面設計をAIと相談  
- `20.計画分割/`：  
  段階的な完成計画（タスク分割）  
- `30.開発・実装/`：  
  - `mock/`：  
    画面イメージからのモック版  
    - `mock_ptn.N/`：  
      パターンを何個か出してもらう（説明を `CHANGES.md` に記載）  
  - `src/`：  
    開発ソース  
- `40.テスト/`：  
  テスト計画・結果レビュー  
- `進捗表.md`：  
  セッションを跨ぐ進捗ログ（追記式・番号なし＝横断）  

### Git 除外対象（.gitignore の根拠）  

生成物・依存物は追跡しない（下記を `.gitignore` に反映する）:  

- C# → `bin/`, `obj/`  
- PHP → `vendor/`  
- JS/TS → `node_modules/`  
- Python → `__pycache__/`, `.venv/`  

※ 言語確定後に `.gitignore` を入れ替える場合も、`/.trash/` の除外と、`.claude/` を**除外しない**方針（参考として追跡する）は維持すること。  

### 進捗表.md の追記フォーマット  

```
## YYYY-MM-DD フェーズ:<フェーズ名>
- 決めたこと: ...
- 未決/宿題: ...
- 次アクション: ...
```

## 逐次確認を挟まない運用  

依頼書に「私の確認無しで一気に行ってください」等の指示がある場合、**逐次「これでいいですか？」と聞かず一気に進める**。ただしこれはAIの振る舞いの話であり、Claude Code のパーミッション（ファイル書込・コマンド実行の許可ゲート）とは別レイヤー。編集確認プロンプトは `.claude/settings.json` の `"defaultMode": "acceptEdits"` で抑止済み（Bash系の確認は安全のため残す）。  

## 上位ポリシー（必ず参照）  

親ディレクトリ `C:\claude\CLAUDE.md` にプロジェクト共通ポリシーがあり、本ワークスペースにも自動適用される。要点のみ再掲:  

- **機密ファイル**: `.env` / `.env.local` / `.env.production` / `.env.staging` の作成・編集は禁止。必要時は `.env.sample`（ダミー値）で代替。`~/.ssh/`・`~/.aws/`・`*.pem`・`*.key` への読み書きも禁止。  
- **削除の扱い**: `rm -rf` 等の直接削除は禁止。削除対象はプロジェクトルート直下の `.trash/` へ `{ベース名}.{YYYYMMDD_HHMMSS}.{拡張子}` 形式にリネームして `mv` で退避。初回作成時のみ `.gitignore` への `/.trash/` 追記を提案する。  
- 詳細な手順・通知フォーマットは `C:\claude\CLAUDE.md` を参照。
